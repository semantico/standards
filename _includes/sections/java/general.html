General Guidelines

In general, we follow the guidelines set out by Sun here. We've decided to adopt Sun's Java Code Conventions because they are sensible, comprehensive and widely used.

However, there are (and will be further) alterations that are specified by Semantico within this document, so you're best bet is to always look here.

If you can't find what you're looking for in this document, always assume that you should use the Sun standards described above.

Indentation

For all code languages, we require indentation to be done via soft tabs (using the space character). Hitting Tab in your text editor shall be equivalent to four spaces.

Line length

Due to expanding screen sizes since the Sun coding standards doc was put together, we operate using a line length of 120 chars (as opposed to 80).

Reformatting of Code

If you find code that's formatted incorrectly, it's definitely advisable to use your chosen IDE to optimize the formatting and then check the changes in, without doing any other changes. Mixing code re-formatting and actual code deltas leads to poor visibility of important changes to the underlying code.

JavaDocs and Comments

Source files should be accompanied by JavaDocs for the class and all methods, especially public methods.

Comments should be added where code usage is not immediately obvious, or some business rule is being implemented.

Commented out Code

Commented out code Commented out code can lead to confusion (see http://nedbatchelder.com/text/deleting-code.html for more detail). Avoid checking commented out code into the version control system. If you comment out code for a short-term experiment, comment it with "//-" so that everyone can see that's what it is, should it accidentally be checked in.
